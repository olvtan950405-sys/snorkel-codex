# Task Explanation — Repair a WireGuard peer-access reconciler

_Category: Security. Reviewer-facing; the agent sees the concise instruction and `/app/docs/access-policy-spec.md`._

## Difficulty Explanation

`tunnelguard` is the enforcement boundary between a zero-trust access policy and the WireGuard and nftables configuration deployed on an edge gateway. It joins enabled peers, address pools, reservations, memberships, routes and emergency grants from SQLite with group and service policy from YAML and per-peer key events from an HTTP API. The shipped Rust program completes successfully and emits plausible configuration and audit files, but its decisions rely on textual address-prefix checks, allow-first group handling, pre-normalization duplicate detection, substring-based event matching and no genuine allocation or route-overlap model. Repair therefore requires coordinating numeric IPv4 and IPv6 semantics, mapped-address normalization, deny and emergency precedence, half-open time windows, compromise and rotation ordering, deterministic address allocation, route authorization, cross-peer conflict precedence and canonical multi-file output. These rules compound: an implementation may classify key history correctly yet still advertise an unauthorized overlapping route, or allocate a valid-looking address while overlooking that its mapped form duplicates another peer.

## Solution Explanation

The oracle validates the policy and database evidence, converts IPv4-mapped IPv6 values before every comparison, masks CIDRs to network addresses and processes enabled peers in byte-sorted ID order. It combines all group allows and removes the union of all denies, then adds only emergency grants active at the fixed evaluation instant. It queries each enabled peer exactly once, validates and time-filters the returned events, and replays compromise before rotation at a shared timestamp to distinguish revoked, staged and inconsistent key states. Configured addresses are validated against numeric pool membership and reservations; invalid addresses receive the lowest usable unreserved address not assigned elsewhere. Only peers surviving higher-priority key and access verdicts have their routes normalized and checked against effectively allowed service networks, after which earlier active peers retain precedence over later overlapping routes. The implementation replaces the output tree, emits enforcement only for active peers and writes a recursively canonical audit whose digest commits to the complete ordered peer array.

## Verification Explanation

The verifier rebuilds the submitted Rust project and drives only `/app/bin/tunnelguard` with explicit temporary database, policy, output and API locations. Its HTTP fixture server serves test-owned event evidence and records requests, proving that disabled peers are omitted and that every enabled peer is queried exactly once. Independent SQLite and policy mutations exercise group-deny precedence, inclusive emergency start and exclusive expiry, deterministic replacement-address allocation, IPv4-mapped duplicate detection, numeric route overlap with first-peer precedence, route containment, future compromise events and inconsistent rotations. Separate checks validate all shipped temporal and key verdicts, exclude rejected peers from both enforcement formats, recompute the peer-array SHA-256, compare canonical JSON bytes, normalize equivalent CIDR spellings and require all four artifacts to remain byte-identical across repeated runs. Because the verifier changes addresses, routes, memberships, policy and event histories outside `/app`, fixture-specific verdicts or precomputed output cannot satisfy the suite.
