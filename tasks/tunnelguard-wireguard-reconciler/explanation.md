# Task Explanation — Close authorization bypasses in a WireGuard security gate

_Category: Security. Reviewer-facing; the agent sees the concise instruction and `/app/docs/access-policy-spec.md`._

## Difficulty Explanation

`tunnelguard` is the authorization boundary between a zero-trust access policy and the WireGuard and nftables configuration deployed on an edge gateway. The seeded gate emits convincing enforcement files while permitting several security bypasses: a mapped address can evade duplicate detection, a group allow can override an explicit deny, compromised or stale keys can remain active, and an unauthorized or overlapping route can redirect protected traffic. Closing those vulnerabilities requires joining enabled peers, address pools, reservations, memberships, routes and emergency grants from SQLite with policy from YAML and key evidence from an HTTP API. Numeric IPv4 and IPv6 identity, deny and emergency precedence, half-open time windows, compromise and rotation ordering, safe allocation, route ownership and conflict precedence must agree before a peer is admitted. These controls compound: correct signature history still leaves the gateway exposed if the same peer can claim another service's route, while correct CIDR containment is insufficient if alternate address spelling bypasses identity conflict checks.

## Solution Explanation

The oracle validates all policy and inventory evidence before authorization, converts IPv4-mapped IPv6 identities before family, pool, reservation and conflict checks, masks CIDRs to network addresses and processes enabled peers in byte-sorted ID order. It computes effective access as the union of group allows minus every deny, then adds only emergency grants active at the fixed evaluation instant. Each enabled peer's event stream is queried once, structurally validated, time-filtered and replayed with compromise preceding rotation at a shared timestamp. Configured addresses are admitted only inside their pool and outside reservations; otherwise the lowest safe unassigned address is selected. Normalized configured-address conflicts are resolved before route validation, every remaining route must belong to an effectively authorized service, and earlier admitted peers retain precedence over later overlaps. Only `active` peers reach WireGuard and nftables enforcement; rejected evidence remains visible in a canonical audit whose digest commits to the complete ordered peer array.

## Verification Explanation

The verifier rebuilds the submitted Rust project and drives only `/app/bin/tunnelguard` with explicit temporary database, policy, output and API locations. Its HTTP fixture server serves test-owned event evidence and records requests, proving that disabled peers are omitted and that every enabled peer is queried exactly once. Independent SQLite and policy mutations exercise group-deny precedence, inclusive emergency start and exclusive expiry, deterministic replacement-address allocation, IPv4-mapped duplicate detection, numeric route overlap with first-peer precedence, route containment, future compromise events and inconsistent rotations. Separate checks validate all shipped temporal and key verdicts, exclude rejected peers from both enforcement formats, recompute the peer-array SHA-256, compare canonical JSON bytes, normalize equivalent CIDR spellings and require all four artifacts to remain byte-identical across repeated runs. Because the verifier changes addresses, routes, memberships, policy and event histories outside `/app`, fixture-specific verdicts or precomputed output cannot satisfy the suite.
