# Task Explanation — Repair refund JWT verification and investigate forged approvals

_Category: Security. Reviewer-facing; the agent sees the concise instruction and `/app/docs/refund-security-contract.md`._

## Difficulty Explanation

`refundguard` combines a live authorization boundary with an incident-response workflow. The shipped Express service accepts a decoded JWT payload without verifying its signature, so an attacker can select `none`, alter identity and merchant claims, or manufacture refund scope. Repairing that flaw is only one part of the task: the operator also needs the captured refund ledger loaded into a durable SQLite database, suspicious signing reuse derived with SQL, and every locked npm package version checked against OSV. These concerns interact with strict input handling and reproducibility. CSV quoting must not weaken schema validation, cross-merchant reuse differs from repeated activity by one merchant, npm lockfiles can repeat the same package coordinate, and remote advisory results must be normalized without making the report order depend on network completion or object iteration. Existing output files are evidence, so malformed input or an unavailable advisory service must not destroy them.

## Solution Explanation

The oracle replaces payload decoding with `jsonwebtoken.verify`, supplies an explicit HS256 allowlist plus the required issuer and audience, and validates the remaining identity, merchant, scope, issued-at, and expiry claim types only after cryptographic verification succeeds. The analyzer parses the complete RFC 4180 ledger and validates every row before creating outputs. It writes validated events to a sibling temporary SQLite database with a typed schema and primary key, then uses `GROUP BY`, `HAVING`, `count(DISTINCT merchant)`, and case-normalized algorithm checks to identify the suspicious groups. Package coordinates are collected from the lockfile root and `packages` entries, de-duplicated, and queried in sorted order through the configurable OSV endpoint. Advisory records and signing groups are normalized and sorted before pretty JSON serialization. The completed temporary database and report are renamed into place only after validation and OSV processing succeed.

## Verification Explanation

The verifier rebuilds the submitted TypeScript and drives only `/app/bin/refundguard`. It starts the Express server with a randomized signing key and generates fresh valid, incorrectly signed, unsigned, incomplete, and expired JWTs, preventing acceptance logic from being tied to supplied tokens. For analysis it creates a temporary quoted CSV ledger and npm lockfile, runs a local HTTP server implementing the OSV query shape, records package requests, and returns advisories with missing and present summaries. It inspects the resulting SQLite schema and integer storage, checks SQL-derived cross-merchant and `none` groups, verifies package de-duplication and stable query order, and compares the exact normalized report. Repeated execution must be byte-identical. Independent malformed-header, duplicate-ID, noninteger-amount, invalid-timestamp, and unreachable-OSV cases preseed both destinations and verify that failure preserves those bytes. The decode-only seed and empty analyzer fail these checks; the oracle implements the general contract and satisfies them.
