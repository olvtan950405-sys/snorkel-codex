# Task Explanation - Harden WAL evidence recovery against forged crash frames

_Category: Security. Reviewer-facing; the agent sees `instruction.md` and `/app/docs/recovery-contract.md`._

## Difficulty Explanation

This is a hard incident-response and binary-forensics task. The security failure is that a gateway recovery gate can treat forged, corrupt, or merely uncommitted WAL frames as trusted evidence after a crash. The agent must decide which bytes in a detached SQLite WAL are authenticated by the format, then produce a standalone database image and a canonical evidence report. The difficulty is not in asking SQLite to recover the file; that path is forbidden because it can mutate or reinterpret the evidence. The load-bearing details now span several defect classes: SQLite WAL structural fields are big-endian while checksum input word order depends on the WAL magic; the rolling checksum carries state across frames; salts must match the WAL header; frames remain untrusted until a valid commit marker closes the transaction; duplicate page frames inside one committed transaction require last-frame ordering; committed page 1 must be trusted before only two standalone-mode bytes are cleared; 65536-byte page-size sentinel handling must agree between SQLite and WAL headers; a later commit can overwrite or truncate earlier pages; checksum-valid frames after the last commit are not durable database state; fatal failures must clean stale artifacts; and path handling must reject aliases and symlink outputs before touching evidence. Corruption handling is also security-sensitive: the first invalid complete frame stops the scan, and later bytes are ignored even if they resemble valid records. The reviewer run showed that strong agents generally handled the core WAL evidence rules but often missed two report-counter edge cases, so the contract now calls them out explicitly: partial trailing bytes do not increment `frames_scanned`, while a complete frame rejected for invalid commit size increments `frames_scanned` but not `valid_frames`.

## Solution Explanation

The intended trust policy validates command shape and path safety before cleanup, then validates the base database and WAL headers, decodes page size including the 65536 sentinel, chooses checksum word order from the WAL magic, and verifies the WAL header checksum before accepting any frame evidence. It scans only complete frame slots while carrying the two-word rolling checksum through each candidate frame. For every complete frame, it validates salts, nonzero page number, checksum, and commit-size constraints before treating the frame as trusted. Valid frames are accumulated in a transaction buffer, and only when a valid nonzero commit-size marker is reached are those frames applied to the working database image in order, so repeated pages in one transaction and later transactions both resolve to the last committed page image. Application is page-based: write each committed page image at `(page_number - 1) * page_size`, extend with zero-filled pages when required, and resize the image exactly to the commit's declared database page count, supporting both growth and truncation. At the end, discard any valid-but-uncommitted tail, apply committed page 1 normally and then clear only SQLite page-1 bytes 18 and 19, compute the SHA-256 digest, and atomically write both the recovered database and the compact ordered JSON report. Fatal evidence errors must return nonzero and leave no output artifacts, while nonfatal tail corruption after at least one valid commit should still recover the last durable snapshot and report the correct stop reason and counters.

## Verification Explanation

The verifier is black-box and independent of the shipped gateway fixture. Each run creates fresh SQLite histories with randomized content, captures multiple standalone snapshots, and constructs WAL files with an independent Python evidence generator. The tests compare exact recovered database bytes against the expected trusted snapshot, verify SQLite integrity, and check canonical JSON serialization byte-for-byte. Coverage includes both checksum byte orders, multiple committed transactions, repeated page frames inside one transaction, committed page-1 handling, later page images winning over earlier ones, valid uncommitted tails, partial trailing frames, checksum/salt/page-number corruption, invalid commit-size rejection, commit-time database truncation, malformed WAL headers, malformed base databases with stale artifact cleanup, mismatched page sizes, no-commit WALs, 1 KiB, 4 KiB, and 65536-byte page sizes, input immutability, independent input/output paths, rejected output/report aliasing, rejected symlink outputs, and rejected trailing CLI arguments. Randomized values, salts, page layouts, frame counts, and output digests make fixture hardcoding brittle. The seeded behavior intentionally fails across validation, data handling, counter ordering, path safety, artifact cleanup, and transaction trust, while the oracle passes the full suite; reviewer telemetry showed at least one frontier agent passing the earlier suite, and these additions preserve determinism while substantially broadening the repair surface.
