# Harden WAL evidence recovery against forged crash frames

During an incident on the telemetry gateway, power was cut while SQLite was checkpointing. The base image at `/app/data/gateway.db` is older than the detached WAL at `/app/data/gateway.db-wal`, and that WAL may include attacker-supplied trailing bytes from a copied crash directory. The current recovery gate accepts bytes that have not been proven durable, which is an evidence-integrity vulnerability: forged, uncommitted, or corrupt WAL frames can contaminate the database snapshot that security later treats as fact.

Close that vulnerability in `/app/bin/walrescue recover --db <path> --wal <path> --out <path> --report <path>` by enforcing `/app/docs/recovery-contract.md` for the supplied gateway evidence and for other database/WAL pairs in the same supported format. Security needs one standalone database image containing only bytes authenticated by SQLite's WAL framing, salts, transaction boundaries, and rolling checksums, plus a compact JSON evidence report that can be cited by digest. Do not invoke SQLite's own WAL recovery, attach or rename the WAL beside the output, or modify either input file. The accepted output is the last intact committed snapshot only; checksum-valid frames after the final commit and every byte after the first corrupt or partial frame are evidence, not durable database state.

The trust decision is intentionally byte-level. Validate the base SQLite header, the WAL magic and format version, page-size agreement, and the WAL header checksum before trusting any frame. WAL structural integers are big-endian, but the magic chooses the byte order for checksum input words, and the two-word checksum state must roll from the header through every accepted frame with uint32 wraparound. Each frame must repeat both salts, name a nonzero page, and match its stored checksum. Keep frames quarantined until a valid nonzero commit-size marker authenticates the whole transaction; only then may those page images update the standalone database, with later committed pages winning and commit sizes allowed to truncate the image.

The report is part of the evidence chain, so its counters must be exact. Stop at the first corrupt complete frame or at a trailing partial frame, and ignore everything later. A partial trailing fragment is not a complete frame slot and must not increase `frames_scanned`. A complete frame rejected for invalid commit size increases `frames_scanned` but not `valid_frames`. Checksum-valid frames after the last commit increase `valid_frames`, not `committed_frames`, and must not affect the recovered database. On fatal evidence errors, including malformed headers, page-size mismatch, or no valid commit, return nonzero and leave no output database or report behind.

Treat command-line and path handling as part of the security boundary. Reject missing flags, unknown flags, or trailing positional arguments. The output database and report paths must be distinct from each other and from the input evidence paths, and existing output/report symlinks must fail closed before anything is removed or written. Existing stale output artifacts from a failed recovery must not survive. The 65536-byte SQLite page-size sentinel, committed page-1 images, repeated page frames inside the same transaction, and later commit-time truncation are all in scope.
