# Task Explanation — Repair TUF repository rotation and delegated trust reconciliation

_Category: Security. Reviewer-facing; the agent sees only the instruction and the contract under `/app/docs`._

## Difficulty Explanation

`repoguard` is the gate between an offline update repository and deployment. It authenticates a TUF-style root-key rotation, walks the timestamp/snapshot/targets chain, consults monotonic versions in SQLite, resolves delegated target ownership, verifies release bytes, and writes the authorization or quarantine state used by the mirror. The shipped Rust implementation does not crash or emit obviously malformed output. Instead, several independent trust failures compose: duplicate signatures satisfy thresholds, a new root authorizes itself without the old root, equal versions are classified as rollback, timestamp length and version commitments are skipped, state advances before the repository has been authenticated, and only top-level targets are reconciled. A plausible report can therefore authorize content using an invalid chain or permanently record a repository that should have been rejected. Repair requires understanding signature authority, canonical signed bytes, metadata containment, rollback boundaries, ordered terminating delegations, filesystem reconciliation, and transactional persistence together.

## Solution Explanation

The oracle first distinguishes a real one-version root rotation from an idempotent same-root rerun. A rotation must reach the old threshold using old authorities and the new threshold using new authorities; every threshold is a set of distinct authorized key IDs whose Ed25519 signatures verify over canonical `signed` bytes. It then validates type, expiry, monotonic version, and all three fields of each parent descriptor before authenticating delegated roles from the top-level delegation keys. Target ownership is resolved in contract order: top-level first, then matching delegations, with an absent target in a terminating role stopping fallback. The union of in-scope descriptors and physical files produces trusted, mismatch, missing, and unclaimed verdicts from actual bytes. Only after global validation does the implementation transactionally advance SQLite and replace the trusted root. Finally it replaces the output tree with canonical per-target artifacts, a canonical aggregate report, and a deterministic audit.

## Verification Explanation

The verifier recompiles the agent's Rust source and drives only the public CLI with explicit temporary repository, state, and output paths. It checks all four target outcomes in the seeded repository, including the load-bearing case where a later broad plugin role contains a valid descriptor that a prior terminating private role must block. Expected target facts are recomputed from current bytes. Separate adversarial tests duplicate one valid signature, corrupt a signature, alter a committed metadata file's stored length without changing its JSON meaning, and place a persisted role ahead of repository metadata. Each global failure must preserve both state stores and replace stale output with only a canonical invalid report. Successful tests check byte-canonical reports and enforcement records, exact audit derivation, custom path handling, equal-version idempotence, and repeat-run byte stability. The shipped implementation fails across threshold, descriptor, delegation, persistence, output, and idempotence tests; the oracle passes the suite.
