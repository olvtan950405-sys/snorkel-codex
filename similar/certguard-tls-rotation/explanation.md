# Task Explanation — Repair Go certguard TLS chain validation, revocation and rotation

_Category: System Administration. Reviewer-facing; lives at the task root, not shipped to the agent (which sees only `instruction.md` and `docs/cert-spec.md`)._

## Difficulty Explanation

`certguard` is the tool a sysadmin runs to keep an edge host's TLS services trustworthy: it reads the certificate inventory, the trust store, the per-service bindings and an evaluation policy, validates the chain behind each service as of a fixed instant, consults a CRL service, and emits the remediation — an nginx snippet rotating a service onto a valid replacement, a systemd drop-in disabling one with no usable certificate, a JSON trust report and a Markdown audit. The difficulty is that it never errors: every run writes a complete, deployable-looking config, but the trust decisions are wrong, so an operator would keep serving a revoked or un-rooted certificate under an audit note that calls it healthy, or disable a service that was actually fine. Four half-finished modules cause it — a validator that checks only the leaf's dates and skips path-length, extended-key-usage and name-constraint enforcement and treats a wildcard SAN as a plain suffix; a revocation client that ignores CRL currency and removeFromCRL and compares serials as text; a selector that grabs the first replacement rather than the best; and a writer that emits the artifacts at the wrong paths. The faults are independent, so fixing any three still fails the byte-exact grader.

## Solution Explanation

This is fidelity to the shipped contract (`docs/cert-spec.md`), not reverse-engineering. Build each chain by linking `authority_key_id` to `subject_key_id` up to a trusted anchor, then validate it as of the policy instant: every non-anchor certificate inside its validity window, path length honored, `serverAuth` chaining, and the service identity matched by an exact or single-label wildcard SAN and by the permitted/excluded name-constraint subtrees. Consult the revocation service once per issuer, failing closed when the CRL window does not cover the instant, ignoring `removeFromCRL` entries and comparing serials by integer value. Classify each service — compliant, or rotate onto the best replacement (latest expiry, then largest serial, then fingerprint), or block when nothing valid covers the identity — and derive the certificate fingerprint as the SHA-256 of a canonical field encoding over a normalized serial. Emit the nginx snippet, systemd drop-in, canonically serialized JSON report and Markdown audit at the exact paths, and rebuild so `/app/bin/certguard` reflects the edit. Every value comes from the inputs; identical inventory and CRLs give byte-identical output, so the run is deterministic, idempotent and auditable.

## Verification Explanation

The grader is black-box: it mints its own TLS inventories and CRL sets, serves the CRLs from a mock that records which issuers were queried, runs the compiled binary, and compares every artifact against an independent Python reimplementation of the whole policy — canonical JSON byte-for-byte, the text artifacts by strict semantic parse. Hard-coding is blocked by deterministically generating twenty fresh inventories whose expected output is recomputed from their bytes, plus a semantic-change test that must move the selected replacement when an expiry changes, and a repeat-run determinism test. Load-bearing details abound: a serial revoked in a different numeric form than the certificate carries; a stale CRL that must fail closed; a `removeFromCRL` entry that must un-revoke; a wildcard that must reject a two-label host; and a pathlen-0 intermediate with a CA beneath it. Each of the four faults alone breaks the golden output. In the container the oracle passes all twenty-six checks while the shipped, half-finished tool fails the great majority of them.
