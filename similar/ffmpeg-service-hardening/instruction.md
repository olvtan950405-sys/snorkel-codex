Our transcode workstation runs a handful of FFmpeg services and I need its package and service configuration brought into a hardened state. The `ffguard` tool in `/app` is meant to do that, but it was left half-finished during a migration and its output can't be trusted yet.

`ffguard` reads the host-state inventory in `/app/data/host_state.db` — the installed packages, the binaries they own, and the systemd units that run them — and checks every FFmpeg service against the OSV advisory API at `OSV_API_BASE`. For each service it has to resolve the package version behind the unit's executable, then, from the advisories that actually apply to that installed version, decide whether the package can be pinned forward to a fixed release, whether the service must be blocked because nothing safe exists or the binary can't be identified, or whether it is already clean. It then writes the APT pin files, the systemd hardening drop-ins, a JSON report, and a Markdown audit note.

Version comparisons follow Debian ordering, and the exact inventory schema, advisory-range semantics, decision rules, and output paths and formats are specified in `/app/docs/hardening-spec.md`. Make `ffguard` produce the correct artifacts for whatever host snapshot and advisory data it is pointed at, and keep it runnable as `/app/bin/ffguard`.
