# Task Explanation — Harden an npm Lockfile Against OSV Advisories

_Category: Security. Reviewer-facing; lives at the task root, not shipped to the agent (which sees only `instruction.md` and `docs/remediation-spec.md`)._

## Difficulty Explanation

`npmguard` is a supply-chain gate an engineer runs to keep a service's committed npm lockfile free of known-vulnerable dependencies: it reads the lockfile and an offline registry snapshot, checks every production package against an OSV advisory feed, and emits a remediation plan — upgrade to a safe version, or quarantine what cannot be fixed. The difficulty is that it never errors: it writes a complete, deployable-looking plan on every run, but the decisions are wrong, so an engineer would ship a lockfile that still carries a live CVE while an audit note calls it clean. The plan is only correct if four hard sub-problems are all solved together. Versions must order by real SemVer precedence, where `1.10.0` outranks `1.9.0` and a pre-release sits below its release. Requirement ranges must be matched with the genuine npm grammar — caret (including the narrow `^0.2.x` and `^0.0.x` rules), tilde, hyphen and x-ranges — plus the notorious pre-release-exclusion rule where `*` and plain comparators never admit a pre-release. Advisory applicability must walk OSV SEMVER intervals, including reintroduction. And the upgrade target must be the *lowest* registry version that clears every advisory while satisfying every dependent's range gathered from anywhere in the tree — a constraint intersection where fixing three of four faults still fails.

## Solution Explanation

This is fidelity to the shipped contract (`remediation-spec.md`), not reverse engineering. Four modules ship incomplete and must be rewritten. `semver.ts` needs full SemVer precedence (numeric identifiers compared as integers, pre-release below release) and a faithful npm range engine that expands each comparator and applies the pre-release-exclusion rule at the comparator-set level. `osv.ts` must actually POST to `/v1/query` with the `npm` ecosystem and walk each advisory's `introduced`/`fixed`/`last_affected` intervals, honouring only SEMVER ranges for the queried npm package and ignoring withdrawn records and foreign decoys. `lockfile.ts` must build the whole model: derive each package name from its `node_modules`-nested path (scopes intact), take the shallowest installed version, mark production scope by excluding dev/optional-only packages, and gather requirement ranges from every node's `dependencies`, not just the root. The already-correct policy then picks the lowest registry version that is higher than installed, unaffected by every advisory, and satisfies every gathered range — distinguishing `no_fix` from `no_safe_version` — and `report.ts` serializes the report as canonical JSON with the overrides map, per-package `blocks/<name>.deny` stanzas (scoped `/` becomes `__`), and the audit note at the exact paths.

## Verification Explanation

The grader is black-box: it runs the compiled tool through a subprocess, serves advisories from a mock OSV server that records what was queried, and compares every artifact against an independent Python reimplementation of SemVer, range satisfaction, OSV containment, the policy and the byte layout. Canonical JSON is matched byte-for-byte; the text artifacts are parsed so content and order are strict while incidental whitespace is not. Hard-coding is defeated by minting fresh lockfiles, registry snapshots and advisory sets from seeded RNG at grade time, with the expected output recomputed from their bytes, and by an anti-cheat pair asserting that identical inputs are byte-stable while a changed advisory fix flips the target. Targeted cases pin each treacherous rule: minimal-target selection, the pre-release exclusion, the `^0.2.x` narrowing, multi-interval reintroduction, constraint gathering from a non-root node, scoped nested names, and the dev/optional scope filter. A 24-project generated corpus must exercise all four decision branches. The shipped tool fails 17 of 18 gradeable tests while the human-written oracle passes them all, and a local 60-seed byte-parity sweep confirms the reference agreement.
