# Repair the signed gateway security-audit endpoint

The partially implemented Express service in `/app` audits nested gateway configuration bundles, but its request authentication, Polars normalization, policy checks, and evidence hashing are all incomplete. Repair the code under `/app/src` and `/app/bin/security-audit.js`; keep the public command `node /app/bin/security-audit.js --port <n>`. At startup the service must read `AUDIT_HMAC_SECRET` as the UTF-8 HMAC key (an unset or empty value is fatal and stderr must contain `AUDIT_HMAC_SECRET is required`) and read the JSON policy at `SECURITY_POLICY_PATH`, defaulting to `/app/data/security-policy.json`. Policy fields are a non-empty string `policy_version`, an array of non-empty strings `allowed_cipher_suites`, an object `minimum_key_bits` whose keys are exactly `EC`, `Ed25519`, and `RSA` and whose values are integers from 0 through 1,000,000 inclusive, a boolean `require_mutual_tls`, and an array of non-empty strings `public_route_paths`; outcomes must be driven by the selected file, not constants, and a missing, malformed, or structurally invalid selected policy is fatal. `GET /healthz` remains unsigned and returns canonical JSON `{"policy_version":"<loaded version>","status":"ok"}` even when a signature header is present. Every JSON response has an `application/json` media type and uses recursively lexicographically sorted object keys, compact UTF-8 JSON, and exactly one trailing newline.

`POST /v1/audit/security-policies` first enforces a 1,048,576-byte raw-body limit while streaming the request: a body over that limit returns 413 with `{"error":"payload_too_large"}` without retaining the excess bytes, while equality is accepted. Within that limit it authenticates the exact bytes received with HMAC-SHA256 before content-encoding, media-type, JSON, or bundle validation. It requires exactly one `X-Audit-Signature` value with the strict form `sha256=` followed by 64 lowercase hexadecimal characters, compares the 32 decoded bytes with `crypto.timingSafeEqual`, and uses the raw bytes rather than `JSON.stringify` output. Missing, duplicate, malformed, uppercase, or incorrect signatures all return HTTP 401 with `{"error":"invalid_signature"}`. After successful authentication, a `Content-Encoding` other than absent or `identity` returns 415 with `{"error":"unsupported_content_encoding"}`; otherwise a missing `Content-Type` or a parsed media type other than `application/json` returns 415 with `{"error":"unsupported_media_type"}` (parameters such as `charset=utf-8` are allowed), a body that is not valid UTF-8 JSON returns 400 with `{"error":"invalid_json"}`, and a JSON value that does not satisfy the bundle schema returns 422 with `{"error":"invalid_bundle"}`. The request body has this shape (arrays shown may contain any number of entries, including zero); `/app/data/example-bundle.json` is the shipped smoke-test example and must remain available:

```json
{
  "bundle_id": "release-2026-06-15",
  "audit_at": "2026-06-15T00:00:00.000Z",
  "gateways": [
    {
      "gateway_id": "gw-east",
      "services": [
        {
          "service_id": "orders",
          "tls": {"minimum_version": "TLSv1.2", "mutual_tls": true, "cipher_suites": ["TLS_AES_128_GCM_SHA256"]},
          "credentials": [{"credential_id": "orders-signing", "key_type": "RSA", "key_bits": 2048, "secret_ref": "vault:orders/signing", "inline": false}],
          "routes": [{"path": "/orders", "methods": ["GET", "POST"], "authentication": "oauth2"}]
        }
      ]
    }
  ]
}
```

The policy version and every configured cipher suite and public route path must also be Unicode-scalar strings; lone UTF-16 surrogates are invalid in all named string fields.

`bundle_id`, gateway/service IDs, credential IDs, cipher suites, route paths, HTTP methods, and secret references are non-empty Unicode-scalar strings (lone UTF-16 surrogates are invalid); strings in the same array are de-duplicated by exact value. `audit_at` uses the exact UTC form `YYYY-MM-DDTHH:mm:ss.sssZ` and must denote a real instant. `key_type` is one of `EC`, `Ed25519`, or `RSA`; `authentication` is one of `api_key`, `mtls`, `none`, or `oauth2`; `key_bits` is an integer from 1 through 1,000,000 inclusive; `tls.minimum_version` and every policy TLS version are one of `TLSv1.0`, `TLSv1.1`, `TLSv1.2`, or `TLSv1.3`; and `tls.mutual_tls` and each credential's `inline` are booleans. Gateway IDs are unique, service IDs are unique within a gateway, credential IDs are unique within a service, and route `path` values are unique within a service. `tls` is a required object and `tls.cipher_suites` a required array, while a missing service `credentials` or `routes`, or a missing route `methods`, is treated as an empty array. Unknown additional object properties are allowed but ignored by normalization and do not enter the evidence digest. Use `nodejs-polars` DataFrames and three separate `explode` operations to flatten gateway `services`, then each service's `credentials` and `routes`, and sort the resulting normalized rows; replacing any required nested normalization dimension with ordinary JavaScript-only flattening does not satisfy this tool-specific task. The normalized evidence inventory has arrays `services`, `credentials`, and `routes`. Service rows contain exactly `gateway_id`, `service_id`, sorted unique `cipher_suites`, boolean `mutual_tls`, and `tls_minimum_version`; credential rows contain exactly `gateway_id`, `service_id`, `credential_id`, `key_type`, integer `key_bits`, `secret_ref`, and boolean `inline`; route rows contain exactly `gateway_id`, `service_id`, `path`, sorted unique `methods`, and `authentication`. All lexicographic ordering in this contract compares Unicode scalar values by code point. Sort services by `(gateway_id, service_id)`, credentials by `(gateway_id, service_id, credential_id)`, and routes by `(gateway_id, service_id, path)`.

Evaluate exactly these five rules. Emit one `CIPHER_SUITE_DEPRECATED`/`high` violation for each unique configured cipher suite absent from the policy allowlist, with that suite as `subject` and evidence `{"allowed_cipher_suites": <sorted allowlist>, "cipher_suite": ...}`. Emit one `MUTUAL_TLS_NOT_ENFORCED`/`high` per service whose `mutual_tls` is `false` when the policy's `require_mutual_tls` is `true`, with subject `tls.mutual_tls` and evidence `{"mutual_tls": false}`. For each credential whose `key_bits` is strictly less than the policy `minimum_key_bits` entry for its `key_type`, emit `WEAK_KEY_SIZE`/`high`, using the credential ID as subject and evidence `{"key_bits": ..., "key_type": ..., "minimum_bits": <policy minimum for that key type>}`; a credential exactly at the minimum is compliant. For each credential whose `inline` is `true`, emit `INLINE_SECRET_EXPOSED`/`critical`, using the credential ID as subject and evidence `{"secret_ref": ...}`. Finally, for each route whose `authentication` is `none` and whose `path` is not one of the policy `public_route_paths`, emit `ROUTE_AUTH_MISSING`/`critical`, using the route path as subject and evidence `{"authentication": "none", "methods": <sorted unique methods>}`. Every violation object has exactly `code`, `evidence`, `gateway_id`, `service_id`, `severity`, and `subject`, and violations are sorted by `(gateway_id, service_id, code, subject)`.

For a valid request, return HTTP 200 with exactly `audit_at`, `bundle_id`, `evidence_digest`, `policy_version`, `service_count`, and `violations`. Build the digest preimage as `{"audit_at", "bundle_id", "inventory", "policy", "violations"}`, where `policy` contains exactly the five loaded policy fields with sorted unique `allowed_cipher_suites` and `public_route_paths`, and the inventory and violation arrays are normalized and sorted as above. Canonically serialize that preimage (recursive key sorting, compact UTF-8 JSON, one trailing newline) and set `evidence_digest` to the lowercase SHA-256 hex digest of those exact bytes. Canonically serialize the response the same way. Consequently, changing only JSON whitespace, input object-key order, gateway/service/credential/route array order, or duplicate cipher-suite/method entries must not change the successful response bytes, while a semantic inventory or policy change must change the digest.
