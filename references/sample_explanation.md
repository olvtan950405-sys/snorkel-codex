# Task Explanation — Repair TypeScript FFmpeg Hardening with OSV Advisories

_Category: System Administration. Reviewer-facing; lives at the task root, not shipped to the agent (which sees only `instruction.md` and `docs/HARDENING_CONTRACT.md`)._

## Difficulty Explanation

`ffguard` is a host-hardening tool a sysadmin runs to keep a workstation's FFmpeg packages patched and its services confined: it reads the apt/systemd inventory, checks each package against an OSV.dev CVE feed, and emits the remediation — an apt pin, a systemd sandbox drop-in, or an order to take the unit offline — plus a Markdown audit note. The difficulty is that it never errors: it writes a complete, deployable-looking config on every run, but the decisions are wrong, so an admin would hold a package at a vulnerable version, drop an override where systemd never scans, or leave a CVE-exposed service running under an audit note that calls it clean. Four regressions cause it — versions ordered as strings instead of Debian order (`5.1.10` sorts under `5.1.9`, the `7:` epoch never stripped); the `introduced` bound dropped, so already-patched packages read as exposed; the inventory reader skipping the binary→package ownership join (the tool's `dpkg -S`), so every service goes unremediated; and the CVE feed stubbed to empty. Output is graded byte-for-byte against a golden digest with nothing hard-coded, and the faults compound (the empty inventory hides the rest), so correcting three of four still fails.

## Solution Explanation

This is configuration fidelity to the runbook (`HARDENING_CONTRACT.md`), not reverse-engineering. Read the whole inventory including the `binary` ownership table — the `dpkg -S` chain tying a unit's `ExecStart` to its package — and the `hardened` flag; load the CVE feed from the snapshot; order versions the Debian way (epoch, then upstream, then revision as numeric fields, so `7:5.1.10` > `7:5.1.4`) and count a package vulnerable only when `introduced ≤ version < fixed`. Then apply the policy per enabled ffmpeg/ffprobe unit — `clean`, `pin` (apt pin at `apt/preferences.d/<pkg>.pref` to the greatest fix), `harden` (systemd drop-in at `systemd/<unit>.d/override.conf` with the standard confinement directives), or `block` — every value derived from the inputs. The plan is float-free configuration-as-data serialized canonically, so identical host state and advisories give byte-identical output and a stable golden digest; the run is idempotent and auditable.

## Verification Explanation

The grader is black-box: it runs the TypeScript through `tsx` (no build, nothing pre-staged), then checks the emitted config against an independent Python reimplementation of the whole policy — byte-canonical, digest-matched to the golden, and confirmed by the `verify` tool printing `OK`. Hard-coding is blocked by synthesizing fresh host and advisory fixtures at grade time (expected output recomputed from their bytes, digest unlike the shipped golden) and an advisory-edit test that must flip a unit from `harden` to `pin` at run time. The shipped host is load-bearing — pin target `7:5.1.10`, and a clean case only because `6.10 ≥ 6.9`, both of which a string sort gets wrong; the mislabelled `apt/<pkg>.pref` and `systemd/<unit>.conf` paths are asserted never written; and an out-of-scope x264 encoder and a disabled libav unit guard the scope filter. Each of the four faults alone breaks the golden, and the shipped tool fails 11 of the 15 tests while the oracle passes all 15.
